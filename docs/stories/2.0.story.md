## User Story

Como sistema SmartPreço,  
quero um motor de cálculo financeiro implementado como funções TypeScript puras,  
para que qualquer módulo (React hooks, API routes, testes) possa calcular viabilidade sem depender de UI ou banco de dados.

---

## Contexto Técnico

Este é o **coração do produto**. O motor de cálculo é isolado de propósito: sem React, sem Supabase, sem Next.js. É TypeScript puro que pode ser executado em qualquer ambiente — browser, Node.js server, worker.

**Princípio fundamental:** funções puras = entrada → saída determinística, zero side effects, 100% testável.

**Precedência de implementação:**  
Esta story PRECEDE as stories 2.1, 2.2, 2.3 (que constroem a UI) e a story 3.1 (simulador). O `@dev` que implementar 2.1 e 3.1 importará o motor criado aqui — não implementará lógica de cálculo nas stories de UI.

**Fórmulas (fonte: `docs/architecture/architecture.md` § 7):**

```
Custo Total = CMV + taxaML + parcelamento + frete + embalagem + impostos + overhead
Margem %    = (Preço - CustoTotal) / Preço × 100
ROI %       = (Preço - CustoTotal) / CustoTotal × 100
Preço Mínimo Viável = busca binária onde Margem = 0%
Preço Recomendado   = busca binária onde Margem = targetMargin
```

**Taxas ML:** lidas do `ML_FEES` e `ML_INSTALLMENT_FEES` de `mercadolivre.config.ts` — NUNCA hard-coded no motor.

## Acceptance Criteria

- [ ] **AC1:** `src/lib/calculations/costs.ts` com função:
  ```typescript
  calculateCostBreakdown(input: ViabilityInput): CostBreakdown
  ```
  - Calcula cada componente do custo separadamente
  - Taxa ML e parcelamento lidos de `mercadolivre.config.ts`
  - `total` = soma de todos os componentes

- [ ] **AC2:** `src/lib/calculations/profitability.ts` com função:
  ```typescript
  calculateProfitabilityMetrics(
    input: ViabilityInput,
    costs: CostBreakdown
  ): ProfitabilityMetrics
  ```
  - Calcula margem%, ROI%, lucro em R$
  - Calcula `breakEvenPrice` (preço onde lucro = 0)
  - Calcula `minimumViablePrice` (margem = 0%)
  - Calcula `recommendedPrice` (margem = `input.targetMargin`)
  - Busca binária com tolerância ≤ R$0.01 e máx 50 iterações

- [ ] **AC3:** `src/lib/calculations/classifier.ts` com função:
  ```typescript
  classifyViability(marginPercent: number): ViabilityClassification
  ```
  - Usa `VIABILITY_THRESHOLDS` de `mercadolivre.config.ts`
  - ≥ 20% → `'viable'`
  - 10–19.99% → `'attention'`
  - < 10% → `'not_viable'`

- [ ] **AC4:** `src/lib/calculations/simulator.ts` com função:
  ```typescript
  generateScenarios(
    baseInput: ViabilityInput,
    steps?: number,          // default 15
    stepPercent?: number     // default 0.05 (5%)
  ): SimulationScenario[]
  ```
  - Gera array de cenários variando o `salePrice` em torno do preço base
  - Cada `SimulationScenario`: `{ salePrice, profit, marginPercent, roiPercent, classification }`
  - Cenários ordenados do menor para o maior preço

- [ ] **AC5:** `src/lib/calculations/engine.ts` com função orquestradora:
  ```typescript
  calculateViability(input: ViabilityInput): ViabilityResult
  ```
  - Chama `calculateCostBreakdown` → `calculateProfitabilityMetrics` → `classifyViability`
  - Retorna `ViabilityResult` completo incluindo `calculatedAt` (ISO 8601)
  - É a única função que os consumidores externos precisam conhecer

- [ ] **AC6:** `src/lib/calculations/index.ts` exportando:
  - `calculateViability` (export principal)
  - `generateScenarios`
  - Funções internas: `calculateCostBreakdown`, `calculateProfitabilityMetrics`, `classifyViability` (para testes diretos)

- [ ] **AC7:** Testes unitários em `tests/unit/calculations/`:
  - `costs.test.ts` — testes de `calculateCostBreakdown`
  - `profitability.test.ts` — testes de `calculateProfitabilityMetrics`
  - `classifier.test.ts` — testes de `classifyViability`
  - `simulator.test.ts` — testes de `generateScenarios`
  - `engine.test.ts` — testes de integração do `calculateViability`

- [ ] **AC8:** Cobertura de testes ≥ 70% nas functions do motor (conforme NFR08 do PRD)

- [ ] **AC9:** Edge cases testados e tratados:
  - `salePrice = 0` → não dividir por zero, retornar métricas zeradas ou erro claro
  - `productCost > salePrice` → margem negativa calculada corretamente
  - `installments` fora do range 1-12 → lançar erro com mensagem clara
  - `listingType` inválido → TypeScript previne em compile time (sem runtime check)

- [ ] **AC10:** `pnpm test` e `pnpm typecheck` limpos

## Casos de Teste de Referência

O `@dev` deve usar estes casos como base dos testes (valores calculados manualmente):

### Caso 1 — Produto Viável (Clássico, 1x)
```
Input:
  productCost: 45.00
  shippingCost: 0 (Envios ML incluso no tipo de anúncio)
  packagingCost: 2.50
  taxRate: 0.06
  overheadRate: 0.05
  targetMargin: 0.20
  salePrice: 99.90
  listingType: 'classic'
  installments: 1

Expected:
  commission: 99.90 × 0.11 = 10.99
  installmentFee: 0
  tax: 99.90 × 0.06 = 5.99
  overhead: 99.90 × 0.05 = 4.99 (overhead sobre preço de venda)
  totalCost: 45 + 10.99 + 0 + 2.50 + 5.99 + 4.99 = 69.47
  profit: 99.90 - 69.47 = 30.43
  marginPercent: 30.43 / 99.90 × 100 ≈ 30.46%
  classification: 'viable'
```

### Caso 2 — Produto em Atenção (Premium, 6x)
```
Input:
  productCost: 80.00
  packagingCost: 3.00
  taxRate: 0.08
  overheadRate: 0.05
  targetMargin: 0.20
  salePrice: 120.00
  listingType: 'premium'
  installments: 6

Expected:
  commission: 120.00 × 0.17 = 20.40
  installmentFee: 120.00 × 0.0643 = 7.72
  tax: 120.00 × 0.08 = 9.60
  overhead: 120.00 × 0.05 = 6.00
  totalCost: 80 + 20.40 + 7.72 + 3.00 + 9.60 + 6.00 = 126.72
  profit: 120.00 - 126.72 = -6.72 (prejuízo)
  marginPercent: ≈ -5.60%
  classification: 'not_viable'
```

### Caso 3 — Break-even e Preço Recomendado
```
Input igual ao Caso 2, mas salePrice usado apenas como ponto de partida
  minimumViablePrice: deve ser > 126.72 / (1 - 0.17 - 0.0643 - 0.08 - 0.05) ≈ a calcular
  recommendedPrice: onde margem = 20%
```
