# Story 6.1 — Tela de Decisão Integrada

**Epic:** EPIC-06 — Motor de Decisão & Polish  
**Status:** InReview  
**Agente:** @dev  
**Esforço estimado:** L  
**Dependências:** Stories 5.3, 3.1, 2.1–2.3 concluídas

## User Story
Como vendedor, quero ver uma recomendação de preço consolidada com base no meu custo e no mercado, para que eu tome uma decisão de preço com total confiança.

## Acceptance Criteria
- [ ] AC1: Tela de Decisão exibe: preço recomendado pelo simulador + posição no mercado
- [ ] AC2: Três opções de posicionamento: Econômico (abaixo da mediana), Competitivo (alinhado), Premium (acima)
- [ ] AC3: Para cada opção: preço sugerido, margem resultante, posição no mercado
- [ ] AC4: Botão "Adotar este preço" salva o `adopted_price` no SKU via API
- [ ] AC5: Resumo da análise: custo total, margem, mediana do mercado, gap do preço escolhido
- [ ] AC6: Fluxo completo navegável: Calculadora → Simulador → Mercado → Decisão (via FlowTabs)

## Escopo

**IN (incluído nesta story):**
- Hook `useDecisionEngine` compondo dados dos 3 hooks existentes
- Componentes `DecisionPanel` e `PositionOptions`
- Persistência do `adopted_price` via `PUT /api/skus/[id]` (endpoint já existente)
- Integração na tab "Decisão" do `FlowTabs`

**OUT (fora desta story):**
- Criar novos endpoints de API (reutiliza o PUT existente)
- Redesenhar o FlowTabs ou outros módulos
- Notificações push ou e-mail ao adotar preço
- Relatórios ou histórico de preços adotados (pós-MVP)

## Riscos

| Risco | Probabilidade | Impacto | Mitigação |
|-------|-------------|---------|-----------|
| Mercado não carregado ao entrar na tab Decisão | Alta | Alto | Graceful degradation: desabilitar opções com mensagem clara |
| P25/P75 indisponíveis se base < 4 anúncios | Média | Médio | Fallback para preço mínimo viável + aviso ao usuário |
| Regressão no `FlowTabs` ao adicionar conteúdo da tab Decisão | Baixa | Médio | Testar navegação entre tabs no E2E |

### Lógica de Posicionamento
[Source: architecture/architecture.md#7-motor-de-calculo-financeiro]

As três opções devem ser calculadas com base em:
- **Econômico:** preço sugerido = P25 do mercado (ou mínimo viável se P25 < custo total)
- **Competitivo:** preço sugerido = mediana do mercado
- **Premium:** preço sugerido = P75 do mercado

Para cada opção, recalcular `marginPercent` usando `calculateProfitabilityMetrics()` de `src/lib/calculations/profitability.ts`.

### API — Salvar Preço Adotado
[Source: architecture/architecture.md#3-estrutura-de-pastas]

Endpoint existente: `PUT /api/skus/[id]` em `src/app/api/skus/[id]/route.ts`
- Payload: `{ adopted_price: number, status: 'for_sale' }`
- Usar cliente Supabase server: `src/lib/supabase/server.ts`
- RLS garante que apenas o dono do SKU pode atualizar

### Estado Compartilhado

### Issues Identificadas

**[HIGH] margin_percent em sku_calculations diverge do preço adotado**
- Arquivo: `src/lib/supabase/skus.ts:145`
- `margin_percent: result.metrics.marginPercent` usa a margem do `salePrice` original (não do `adoptedPrice`)
- Impacto: dado denormalizado inconsistente — `sale_price` está correto, `margin_percent` pode divergir
- Mitigação atual: `cost_data` e `sale_price` estão corretos, dados podem ser recalculados
- Fix sugerido: recalcular `calculateViability({ ...input, salePrice: adoptedPrice })` e usar suas métricas

**[MEDIUM] `<a href>` no estado de sucesso em vez de `<Link>`**
- Arquivo: `src/components/decisao/DecisionPanel.tsx:57-65`
- Links "Ver SKU" e "Meus SKUs" usam `<a href>` causando full page reload desnecessário
- Fix sugerido: substituir por `import Link from 'next/link'` com `<Link href={...}>`

**[MEDIUM] Formulário sem `<form>` nativa — acessibilidade reduzida**
- Arquivo: `src/components/decisao/DecisionPanel.tsx:157-219`
- Div com inputs sem wrapper `<form onSubmit>` — Enter funciona via `onKeyDown` mas não é semântico
- Impacto: screen readers não identificam o grupo como formulário
- Fix sugerido: envolver inputs em `<form onSubmit={handleAdopt}>`

**[MEDIUM] AC6 — FlowTabs não implementado como especificado**
- A story especificava tab "Decisão" no FlowTabs existente
- Implementação KISS (inline no CostForm) é funcional e defensável para MVP
- Risco: FlowTabs pode ser adicionado no futuro e quebrar a integração atual
- Decisão: aceitar como WAIVED para MVP; registrar como backlog de UX

**[LOW] Ausência de teste para `adoptSku` e `adoptSkuAction`**
- `tests/unit/hooks/decisionEngine.test.ts` testa apenas cálculos puros
- Persistência via Supabase não tem cobertura de teste (mesmo com mock)
- Aceitável para MVP dado que RLS garante segurança no banco
